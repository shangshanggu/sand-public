## Purpose
This guide explains how to populate `reproduced/config/thesis.yml` so that
automation helpers, R scripts, and QA checks share one source of truth. Read it
before running Make targets that depend on configuration. For related material,
see the [documentation index](../README.md).

## File Location & Validation
- The canonical configuration file lives at `reproduced/config/thesis.yml`.
- Run `make validate-config` (or invoke `python scripts/00_setup/validate_config.py`) to verify syntax, required keys, and type constraints before committing edits.
- Store environment-specific secrets outside of version control. This guide only covers paths and parameters that belong in the repository.

## Top-Level Structure
`reproduced/config/thesis.yml` is organized into four primary sections. Each section is required unless otherwise noted.

### 1. `project`
Defines repository-wide metadata and core filesystem layout.

Key expectations:
- `name`, `version`, and `timezone` provide descriptive metadata for logging.
- `paths` enumerates relative directories for data, outputs, and thesis assets. Keep these paths relative to the repository root so containerized workflows remain portable.
- `environment` records the Conda environment, `renv` lockfile, manifest, and Docker image tags used for reproducibility.

### 2. `data`
Tracks raw dataset locations and integrity artifacts.

Key expectations:
- `mode` selects which raw dataset directory is active (`real` or `proxy`).
- `proxy_dir` points to the staged proxy inputs used when `data.mode: proxy`.
- `raw_files` maps descriptive identifiers (e.g., `participants`, `outcomes`) to CSV locations under `data/raw/`.
  These CSVs are optional for the core pipeline; if you only have `list_by_wave.RData`,
  run `make export-raw-csvs` to derive them from Wave 1 for hashing/smoke checks.
- `metadata_manifest` and `checksum_log` point to JSON outputs generated by the integrity scripts.
- `allow_missing_raw` should remain `false` during automation to surface missing files early.
- You can temporarily override the mode with `SAND_DATA_MODE=proxy` when running `make` or `Rscript` commands.
- When running via `make`, you can also override with `DATA_MODE=proxy` (the Makefile propagates the effective mode to `SAND_DATA_MODE` for scripts).

### 3. `chapters`
Controls chapter-specific toggles and analysis dependencies.

Key expectations:
- Each chapter entry (e.g., `chapter4_data_collection`) includes `enabled`, `description`, `scripts_dir`, and `outputs_dir` fields to drive downstream tooling.
- `required_inputs` lists upstream artifacts that must exist before the chapter executes.
- Optional chapter-specific keys (such as `qa_expectations`, `reference_results`, or `target_coefficients`) capture validation thresholds used in QA scripts.
- Disable chapters not yet ready for automation by setting `enabled: false`. Downstream `make` targets should respect this flag.
- `chapter4_data_collection` includes a `spec_parameters` block that enumerates the model combinations generated by `reproduced/analyses/chapter4_data_collection/scripts/01_data_preparation_norms.R`. Adjusting these YAML lists lets you tune the R pipeline without editing code.

### 4. `rsiena`
Centralizes RSiena runtime parameters for Chapter 7 analyses.

Key expectations:
- `use_cached_results`, `cache_strategy`, and `project_seed` define reproducibility posture for long-running estimation jobs.
- `estimation` holds tuning parameters (`n3`, `n2`, `firstg`, `max_iterations`, `target_phase_tolerance`) that were extracted from legacy scripts.
- `diagnostics` directs where convergence summaries are written.
- `parallelization` specifies whether multi-core execution is allowed and how many workers to allocate.

## Update Workflow
1. Edit `reproduced/config/thesis.yml` to reflect new paths or parameters.
2. Run `make validate-config` to ensure the schema remains consistent.
3. Record material changes in a decision or status note so collaborators
   understand why configuration values shifted.
4. If a new chapter or module needs configuration, extend the schema and update this guide accordingly.
