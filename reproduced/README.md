# README

## Why everything lives under `reproduced/`
The goal of this folder is to keep the reproduction workspace tidy. Instead of
hunting through the repository for `config/`, `data/`, `outputs/`, or
`scripts/`, every moving piece now lives under `reproduced/`. Treat it as the
control room for the maintained Chapters 4–7 code. The verified public fast
path covers Chapters 4–6; Chapter 7 is a partial, long-running manual
diagnostic. Chapter 8 is experimental and disabled by default.

Start with [`REPRODUCIBLE_PIPELINE.md`](./REPRODUCIBLE_PIPELINE.md) for the
canonical build contract, quick-start commands, and chapter-level guarantees
before diving into the directory details below.

## Quick Start (public proxy workflow)
- Clone this repository.
- Install a current Conda distribution on macOS or Linux; the environment supplies R 4.3.2.
- From the repo root, run `make env` to restore Conda and renv (uses `environment.yml` and `renv.lock`).
- Run `make verify-proxy-quick` to regenerate proxy inputs, execute Chapters
  4–6, and run the 16 structural checks.
- Run `make verify-proxy` only if you deliberately want the optional,
  long-running Chapter 7 diagnostic.
- Inspect outputs under `reproduced/outputs/` and logs under `reproduced/logs/` or `reproduced/outputs/*/logs/`.

Authorised users of the protected analysis bundle should follow “Real data
onboarding” below and use `make verify-real`.

## How outputs are organised
Pipeline runs write everything into `reproduced/outputs/` so each chapter’s
results stay grouped together:

| Folder | Key artefacts |
| --- | --- |
| `reproduced/outputs/chapter4/` | Longitudinal extracts, QA manifests, and network arrays exported from Chapter 4 scripts. |
| `reproduced/outputs/chapter5/` | NAM-ready CSVs, diagnostic comparisons, and benchmark checksums. |
| `reproduced/outputs/chapter6/` | Injunctive norm time series, shift summaries, and supporting plots. |
| `reproduced/outputs/chapter7/` | SAOM inputs, regenerated coefficient tables/plots, validation manifests, diagnostic logs, and cached baseline fits. |
| `reproduced/outputs/chapter8/` | Reserved for intervention staging artefacts and scenario manifests. |
| `reproduced/logs/` | JSON execution logs and hash manifests generated by setup helpers. |

If you delete these folders, running the relevant `make` target will rebuild the
contents using the data and scripts staged in this workspace.

## Meet the configuration file
`reproduced/config/thesis.yml` tells every script where to find inputs, where to
write outputs, and which chapters are enabled. When you move files or rename
folders, update this YAML first and then run:

```bash
make validate-config
```

This executes `reproduced/scripts/00_setup/validate_config.py`, which confirms
that the YAML structure is still valid and all referenced folders exist.

## Makefile cheat sheet
Run these commands from the repository root:

```bash
make env              # Refreshes Conda via environment.yml and renv via renv.lock
make env-conda        # Only touch the Conda environment
make env-renv         # Only restore the R packages snapshot
make validate-config  # Sanity-check reproduced/config/thesis.yml
make proxy-data       # Generate realistic proxy inputs under reproduced/data/proxy/
make chapter4         # Reproduce Chapter 4 data exports and QA checks
make chapter5         # Rebuilds Chapter 4 exports and the entire Chapter 5 NAM pipeline
make chapter6         # Recreates approval datasets, metrics, and figures for Chapter 6
make chapter7         # Installs/updates RSiena and regenerates the SAOM baseline outputs
make verify-proxy-quick # Fresh proxy build and structural checks for Chapters 4–6
make verify-proxy     # Optional long Chapter 7 proxy diagnostic
make verify-real      # Protected-data build with empirical benchmark checks
make visualise        # Generate network visualisations from proxy data
make help             # Lists every available target with a one-line description
```

## Resetting and rebuilding outputs
To verify the project is reproducible end to end, follow this cycle:

1. **Back up the current artefacts** (optional but recommended):
   ```bash
   mkdir -p /tmp/reproduced_backups
   cp -R reproduced/outputs /tmp/reproduced_backups/outputs_commit
   ```
2. **Clear generated results** (only after backing them up):
   ```bash
   rm -rf reproduced/outputs/chapter5 reproduced/outputs/chapter6 reproduced/outputs/chapter7
   rm -f reproduced/logs/*.json
   ```
3. **Regenerate the chapters**:
   ```bash
   make chapter5
   make chapter6
   make chapter7
   ```
4. **Compare fresh outputs to your backup**:
   ```bash
   diff -rq reproduced/outputs /tmp/reproduced_backups/outputs_commit
   ```
   No output means the rebuild matches your saved comparison copy. If differences
   appear, inspect the JSON logs in `reproduced/logs/` to understand
   what changed before deciding whether to accept or investigate the delta.

## Network Visualisation

The `scripts/visualisation/` directory contains scripts for rendering the
longitudinal friendship networks. These work with proxy data (or real data if
available) and produce publication-quality figures:

```bash
make proxy-data    # Generate synthetic network data
make visualise     # Render all network plots
```

## Real data onboarding
- Required protected file: `reproduced/data/raw/list_by_wave.RData` (the analysis reads this directly).
- Optional CSVs: `reproduced/data/raw/participants.csv` and `reproduced/data/raw/outcomes.csv` are used for hashing/smoke tests; derive them from Wave 1 with `make export-raw-csvs` if you only have the RData export.
- Source: an analysis-facing wave bundle assembled from controlled institutional
  REDCap CSV exports (not tracked here). Keep approved provenance notes and
  hashes outside the public repository.
- Proxy inputs live under `reproduced/data/proxy/` and are only used when `data.mode: proxy` (or `SAND_DATA_MODE=proxy`) is set. Generate them explicitly with `make proxy-data`.
- If you have checksums for the real exports, record them in `reproduced/logs/data_hashes.json` via `make verify-real` (or keep a copy alongside the data). Missing optional CSVs will be reported by the checksum manifest, while missing `list_by_wave.RData` will stop `make chapter4`.

## Logs, QA, and verification pointers
- Top-level logs: `reproduced/logs/` (e.g., `run_all_history.json`, `validate-config/`, `chapter*/`).
- Chapter-specific logs and manifests: `reproduced/outputs/chapter*/logs/` plus chapter data/figures/tables under the same parent.
- Verification: `make verify-proxy-quick` is the verified Chapters 4–6 public
  gate. `make verify-proxy` adds the partial Chapter 7 diagnostic;
  `make verify-real` enforces protected-data benchmarks and convergence checks.
  `make verify` dispatches according to the configured data mode.
- Chapter 8 staging artifacts: `reproduced/outputs/chapter8/logs/` (manifest snapshots, staging summaries).
